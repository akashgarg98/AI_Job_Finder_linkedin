# Product Manager Q&A Document - Job Finder AI Portfolio

**Q: Can you walk me through the overall lifecycle and architectural pivots of your Job Finder AI application?**
**A:** We initially launched the MVP as a Next.js / Express web platform built entirely around generic PDF scraping and basic deterministic keyword indexing algorithms. However, as we expanded the internal capability scenarios, I identified multiple critical latency and privacy gaps. Thus, the intelligence layer evolved dynamically across major architectural sprints:
1. Completely deprecating the backend string counting module by shifting definitively towards a **Google Gemini LLM** (using the new v5.x SDK) for extreme-accuracy semantic scoring logic inside a prompted HR constraint.
2. Formally abstracting the entire PDF extraction protocol away from the backend pipeline in favor of a **Zero-Transfer Client-Size Architecture**, executing the heavy `pdfjs-dist` parsing nodes completely offline in the user's DOM memory sandbox. This guarantees peak PII safety, as user files *never internally transmit*.
3. Overhauling the generic data simulation algorithms to mimic truly organic Apify payloads natively by randomizing company targets dynamically inside a rigid 24-hour relevance filter (`&f_TPR=r86400`).

**Q: What significant technical roadblock did you overcome from a distinct Product perspective during scale?**
**A:** Our most punishing hurdle was the primitive REST payload transport streams out of the Next.js Client modules. I found that Node.js streams dropping buffers silently threw horrific UX failure states natively for the end user every time the backend node crashed. 
As the PM leading structure, rather than trying to endlessly patch the rigid boundaries of `multipart/form-data` payload protocols, I championed a permanent paradigm pivot. *We just physically stopped sending files completely*.
Instead, we built `pdfjs-dist` deep into the React `onChange` file handler loop. So the very moment the user actively drops a file, the browser locally rips it into text internally using Web Workers, and we ONLY ever forward the 100% clean UTF-8 text bundle directly to our backend server. This architecturally eliminated 100% of all Node crash bugs and increased application speed dramatically. 
Furthermore, when the dependency’s Web Worker CDN began throwing Mixed-Content CORS conflicts explicitly inside `localhost` sandboxes, I mandated the engineering layer offline the entire worker logic into the Next.js `public/` directory exclusively to eliminate all network-bound failures for good.

**Q: How does the AI scoring model actually evaluate candidate alignment securely?**
**A:** Previously, it just searched against generic expressions blindly using Regex scripts; an incredibly useless metric. We removed that completely, locking in the `GoogleGenAI` module to proxy candidates intelligently through `gemini-2.5-flash`. The backend issues prompts explicitly trapping Gemini inside HR Generalist constraints natively asking it to process and return a completely isolated numeric array item `(0-100)` measuring distinct context, dynamic relevance, and implicit experience overlays. The Express router caches that pure numeric code array independently, and seamlessly maps it cleanly onto the Next.js visual feedback grid automatically applying standard (Green/Yellow/Red) alert logic internally.

**Q: If I wanted to hook this exact system up right now to a live APIfy Account trigger over the simulations, how would that loop behave natively?**
**A:** Extremely differently structurally. Because real-time targeted Apify scripts routinely buffer between 60 seconds to up to 10 full minutes dynamically, our internal Next.js fetch components would trigger complete application Server-Timeout loops natively since they block synchronously. We are simulating instant generation now explicitly for rapid product-validation metrics. To migrate immediately to a heavy live external scrape layer securely, I would reconfigure the specific application architecture into a Webhook listener relay interface. Apify would trigger its own detached process, and upon termination, it would hit our backend API webhook listener, natively populating the array database seamlessly. Then we would optionally blast the feed via WebSockets to visually paint the data directly to the active user's screen in real-time.
